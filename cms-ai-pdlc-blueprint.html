Part III

The End-to-End Workflow at a Glance

The following six steps take capability from raw product requirements all the way to a merged, tested, deployed service — with Claude Code as an informed collaborator at every stage. Reference document.

Step 1

Author CLAUDE.md Before Writing Any Code

Lock in the tech stack, folder structure, hard rules, and workflow commands so AI starts every session informed.

Step 2

Translate Product Requirements into Scoped Work

Use Claude to decompose PRDs into bounded, testable units of work. Validate scope with the team before moving to tasks.

Step 3

Generate and Assign Git Project Tasks

Convert scoped work into discrete, trackable issues. Include acceptance criteria and dependencies in every task.

Step 4

Implement Each Task with Claude as Pair Programmer

Leverage CLAUDE.md guardrails during implementation. Claude proposes a plan first, then implements the smallest viable change.

Step 5

Validate and Harden Before Merge

Run the commands defined in CLAUDE.md. Verify that Claude's output respects architecture, async patterns, and test coverage.

Step 6

Ship and Update CLAUDE.md

Merge, deploy, and capture any new patterns or lessons learned back into CLAUDE.md so the next session starts smarter.

Step 7 — Agentic Orchestration

Move from one-session pair programming to coordinated multi-agent pipelines. Claude shifts from responder to goal-seeking orchestrator with bounded tools, explicit gates, and auditable actions.

CLAUDE.md Is The Contract

Pipeline Pattern (Trigger to Audit)

TriggerSignal fires: webhook, schedule, or threshold breach.
EnrichWorker pulls contract + job context via approved MCP tools.
ClassifyCritic scores severity and confidence against schema rules.
GateHigh severity pauses for PM acknowledgment before actions.
Act + LogCreate remediation issues, notify owners, emit audit event.

ERP example: cost overrun event -> contract lookup -> variance summary -> GitHub issues -> PM notification -> Application Insights audit.

Roles + Guardrails

Orchestrator

Breaks goals into subtasks, delegates, aggregates, never implements directly.

Worker

Owns one service or repo boundary and returns structured outputs only.

Critic

Checks policy, contracts, and acceptance criteria before human visibility.

  • Allowed tools only: MCP inventory is explicit, no undeclared APIs.
  • No direct main commits: PR + review gate required.
  • No silent failures: low confidence or errors must escalate.
  • Rollback required: failed runs leave no partial state behind.

Example Orchestration Run: Construction ERP Cost Variance

Event-Driven
Trigger + Goal: JC cost event breaches threshold. Goal: produce a reviewed remediation packet with owner-ready tasks and full audit trace.
MCP Tools: `mcp-vista-finance` (read costs), `mcp-github` (create issues), `mcp-notifications` (Teams), `mcp-appinsights` (audit logs).
Execution: Orchestrator delegates enrichment to worker, critic scores confidence, then opens variance issue set by service label.
Gate + Rollback: High severity pauses for PM acknowledgment. On failure, created tasks are closed and pipeline state is reverted before retry/escalation.

How this extends Steps 1-6: Step 1 adds agentic rules and tool inventory; Step 2 marks automatable vs human-gated requirements; Steps 3-4 split trigger/worker/critic components; Step 5 validates gates, audit, rollback; Step 6 updates rules after every production run.
